# ADR 0003: Redis как персистентная очередь для задач сбора истории

**Статус**: Принято  
**Дата**: 2026-06-14

## Контекст

При обнаружении изменения ставки финансирования или нового символа нужно запустить сбор исторических данных. Сбор может занимать секунды (до 1000 записей на биржу). Нельзя блокировать основной цикл сбора текущих ставок.

Требования:
- Надёжность: задачи не должны потеряться при рестарте API
- Порядок: не критичен, но FIFO предпочтительнее
- Backpressure: ограничение параллелизма (не больше N одновременных запросов к API биржи)
- Простота: минимальные зависимости

## Решение

**Redis List (LPUSH / RPOP)** как персистентная очередь задач.

- Producer (`HistoricalCollectionProducer`): `LPUSH` задачи в `funding_monitor:history_queue`
- Consumer (`HistoricalCollectionBackgroundService`): `RPOP` (блокирующий `BRPOP` с таймаутом)
- Параллелизм: `SemaphoreSlim(MaxConcurrentTasks)` в консьюмере
- Задача: `HistoricalCollectionTask { Symbol, Exchange, From, To, RetryCount }` — JSON сериализация

При ошибке — инкремент `RetryCount` и `LPUSH` обратно (макс 3 попытки).

## Последствия

✅ **Плюсы**:
- Персистентность: выживает при рестарте API/контейнера
- Атомарные операции (LPUSH/RPOP)
- Простая инспекция: `LRANGE funding_monitor:history_queue 0 -1`
- Встроенный TTL не нужен — задачи удаляются после обработки
- Есть в docker-compose, не добавляет новые сервисы

❌ **Минусы**:
- Нет приоритетов (все задачи равны)
- Нет dead-letter queue (после 3 ретраев задача теряется — логируется и дропается)
- Нет видимости прогресса внутри задачи
- При высокой нагрузке Redis может стать bottleneck

## Альтернативы

| Вариант | Причина отказа |
|---------|----------------|
| In-memory Channel / BlockingCollection | Потеря задач при рестарте |
| PostgreSQL таблица как очередь (SKIP LOCKED) | Сложнее, медленнее, нужен polling |
| RabbitMQ | Лишний сервис, overkill |
| Hangfire / Quartz с_job storage | Избыточно для одной очереди |
| Kafka | Оверкилл, не нужен replay/streaming |

## Примечание

Если понадобятся приоритеты, DLQ, метрики — мигрируем на RabbitMQ илиHangfire.
